# 仓库管理系统（WMS）架构提示词

> 与当前仓库代码对齐的架构约束文档，供开发与 AI 协作使用。  
> 技术栈：Vue 3 + Vite + Element Plus + Flask + MySQL（本地双进程开发，无 Docker）。

**默认假设：**

- 单组织、多仓库；商品按编码（`sku_code`）管理；库存精确到「仓库 + 库位 + 编码」
- 入库 / 出库 / 盘点均走单据审批流；库存变更**只**通过 `inventory_service`
- 供应商 / 客户暂为单据上的字符串字段，无独立主数据表

---

## 1. 整体架构

```
┌─────────────────┐     HTTP/JSON      ┌─────────────────┐     SQLAlchemy     ┌─────────────────┐
│   WmsClient     │  ───────────────▶  │   WmsServer     │  ───────────────▶  │  MySQL          │
│ Vue+Vite+EP     │   /api/v1/*        │ Flask + Waitress│   wmsdatabase      │  WmsSQL 管库    │
│ :5173           │                    │ :5000           │                    │                 │
└─────────────────┘                    └─────────────────┘                    └─────────────────┘
```

**基本原则：**

- **前后端分离**：前端只负责 UI、路由、状态与请求封装；后端负责业务逻辑与数据持久化。
- **契约驱动**：接口路径、请求/响应格式、鉴权方式前后端一致；改一端须同步另一端。
- **分层清晰**：前端 API 层集中封装；后端 routes 薄、services 厚、models 只做映射。
- **库存可追溯**：任何数量变动必须留下**单据 + 流水**。
- **配置外置**：数据库、密钥、API 地址、GPIO 均通过环境变量，不提交 `.env`。

**仓库模块：**

| 目录 | 职责 |
|------|------|
| `WmsClient/` | Vue 前端 |
| `WmsServer/` | Flask REST API |
| `WmsSQL/` | `init.sql`、`seed_users.sql`、表结构说明 |

**业务域（已实现）：**

| 域 | 说明 |
|----|------|
| 主数据 | 用户、角色、仓库、库位、商品（编码） |
| 库存 | 现存量、安全库存预警、库存流水 |
| 入库 / 出库 / 盘点 | 统一单据状态机；供应商/客户为字符串 |
| 系统 | JWT 登录、改密、用户管理、健康检查、周期出入库汇总看板 |
| 可选硬件 | 审核通过后 GPIO 脉冲（`GPIO_ENABLED`，默认关） |

### 2.1 技术栈

- Vue 3 + Vite + TypeScript
- Element Plus + `@element-plus/icons-vue`

### 2.3 页面与路由

| path | 权限 |
|------|------|
| `/login` | 公开 |
| `/` 看板 | 登录 |
| `/products`、`/warehouses`、`/users` | **仅 admin** |
| `/inventory` | 登录 |
| `/inbound`、`/outbound`、`/stocktake`（含 new / `:id`） | 登录 |

### 2.4 架构约束

| 规则 | 说明 |
|------|------|
| 路径集中 | API 路径**仅**在 `src/api/paths.ts`；禁止组件手写路径 |
| 请求统一 | 经 `http.ts`；组件只调 `api/*.ts` |
| 环境变量 | `VITE_API_BASE_URL`（默认 `/api/v1`）；Vite 代理 `/api` → `:5000` |
| 单据 UI | 按 `status` 展示标签与按钮（草稿 / 待审 / 已完成 / 已取消） |
| 数量 | 与后端一致，最多 3 位小数 |
| 职责边界 | 前端**不**计算最终库存、**不**连库 |

### 2.5 对 AI 的约束

- 不引入与现有栈冲突的框架。
- 不删除鉴权、错误处理、请求封装，除非明确要求并给出替代方案。
- 改动保持最小必要范围；同时改前后端时同步契约。
- 库存页面只展示后端返回值，不在前端本地加减库存。

## 3. 后端架构（WmsServer）

### 3.1 技术栈

- Flask 2.0+、Flask-SQLAlchemy、PyMySQL
- Flask-Migrate（已接入 extensions，**当前无 migrations 目录**，以 `init.sql` + `init_db.py` 为准）
- PyJWT、Waitress、python-dotenv、Werkzeug（密码哈希）
- API 前缀：`/api/v1`

### 3.2 目录结构

```text
WmsServer/
├── app/
│   ├── __init__.py              # create_app、蓝图注册、seed_data
│   ├── config.py
│   ├── extensions.py            # db, migrate
│   ├── models/                  # 13 张表 ORM（含 order_no_seq）
│   ├── routes/                  # auth, products, warehouses(+locations),
│   │                            # inventory, inbound/outbound/stocktake,
│   │                            # users, dashboard, health
│   ├── services/
│   │   ├── inventory_service.py # ★ 现存量变更唯一入口
│   │   ├── inbound_service.py
│   │   ├── outbound_service.py
│   │   └── stocktake_service.py
│   └── utils/
│       ├── auth.py              # login_required / role_required / JWT
│       ├── response.py
│       ├── exceptions.py
│       ├── order_no.py
│       └── gpio_control.py      # 审核脉冲（可选）
├── init_db.py
├── serve.py                     # Waitress 0.0.0.0:5000
├── wsgi.py
├── requirements.txt
└── .env.example
```

### 3.3 分层架构

```
HTTP Request
     ↓
routes（蓝图）    ← 薄：取参 → 调 service → 返回 JSON
     ↓
services         ← 单据状态机、库存变更、事务边界
     ↓
models           ← 表映射
     ↓
MySQL
```

| 层 | 职责 | 禁止 |
|----|------|------|
| **routes** | 解析 request、鉴权、调 service、返回 JSON | 库存加减、复杂业务规则 |
| **services** | 单据流转、调 `inventory_service`、事务 | 直接依赖 Flask `request` 细节 |
| **models** | ORM 映射 | 跨表库存逻辑 |
| **inventory_service** | **唯一**修改 `inventory.quantity` 并写流水 | 被 routes 绕过 |

### 3.4 认证与权限

- JWT：`Authorization: Bearer <token>`，payload `{sub, role, exp}`，默认 24h
- 角色：`admin` / `warehouse_keeper` / `viewer`
- `role_required`：角色在列表中，或为 `admin` 则放行
- **当前权限差异**（以代码为准）：
  - 主数据写、用户管理：仅 `admin`
  - 入/出库建改提交删除：`admin` / `keeper` / **`viewer`**
  - 入/出库审核驳回取消、盘点写操作：`admin` / `keeper`
  - 盘点：`viewer` 仅可读列表/详情

### 3.5 核心业务规则

#### 库存变更铁律

- `inventory.quantity` **只能**经 `inventory_service.adjust_quantity` 变更。
- 每次变更写入 `inventory_transaction`：`product_id`、`warehouse_id`、`location_id`、`delta_qty`、`balance_after`、`ref_type`、`ref_id`、`operator_id`。
- 出库前校验可用库存，不足 → `409` + `INSUFFICIENT_STOCK`。

#### 单据状态机（入 / 出 / 盘）

| 动作 | 前置 | 结果 |
|------|------|------|
| create | — | `draft` |
| update / delete | `draft` | — |
| submit | `draft` + 有明细 | `submitted` |
| approve | `submitted` | 过账后 **`completed`**（写 `approved_by/at`） |
| reject | `submitted` | 回到 **`draft`** |
| cancel | `draft` / `submitted` | `cancelled` |

说明：枚举中保留 `approved`，但审核过账后对外立即为 `completed`，不持久停留在 `approved`。

过账要点：

- **入库**：`actual_qty > 0` 用 actual，否则 planned；`adjust_quantity(+)`
- **出库**：先逐行校验，再 `adjust_quantity(-)`
- **盘点**：快照 `book_qty`，`diff = counted - book`；diff≠0 才调整；`counted=0` 且 `book>0` 拒绝
- 审核成功可触发 `gpio_control.pulse_on_approve`（需 `GPIO_ENABLED=true`）

#### 单据号

- 格式：`IN` / `OUT` / `ST` + `YYYYMMDD` + 4 位序号（`order_no_seq` + `FOR UPDATE`）

### 3.6 主要 API（前缀 `/api/v1`）

| 资源 | 说明 |
|------|------|
| `GET /health` | 健康检查（无鉴权） |
| `POST /login`、`GET /me`、`POST /me/change-password` | 鉴权 |
| `/products`、`/products/suggest-code` | 商品 CRUD + 编码建议 |
| `/warehouses`、`/locations` | 仓库 / 库位（库位挂在 warehouses 蓝图） |
| `/inventory`、`/inventory/transactions`、`/inventory/alerts` | 库存 |
| `/inbound-orders`、`/outbound-orders`、`/stocktake-orders` | 单据 CRUD + submit/approve/reject/cancel |
| `/users`、`/users/<id>/reset-password` | 用户管理（admin） |
| `GET /dashboard/product-movement` | 看板：周期内按产品汇总入/出库 |

看板接口说明：

- 查询参数：`start_date`、`end_date`（`YYYY-MM-DD`，必填语义上由前端传入）、可选 `warehouse_id`
- 日期按**东八区日历日**解释；与 UTC 存储的 `inventory_transaction.created_at` 换算后再过滤
- 返回每条产品的周期入库合计、出库合计，以及实时当前库存、安全库存
- 前端支持点「查询」刷新，并导出 CSV / Excel（`utils/exportTable.ts`）

### 3.7 环境变量（`.env.example`）

| 变量 | 默认 |
|------|------|
| `MYSQL_*` | `127.0.0.1` / `3306` / `root` / `wmspassword` / `wmsdatabase` |
| `JWT_EXPIRE_HOURS` | `24` |
| `GPIO_ENABLED` | `false` |
| `GPIO_CHIP` / `GPIO_LINE` / `GPIO_PULSE_SECONDS` | `gpiochip1` / `73` / `5` |

### 3.8 新增功能顺序

```
models → services（含 inventory_service）→ routes → 前端 api/paths + views
```

改表时同步：`WmsSQL/init.sql` → `app/models/` →（可选）Flask-Migrate。

## 4. 数据库架构（WmsSQL / MySQL）

### 4.1 连接

```
mysql+pymysql://{USER}:{PASSWORD}@{HOST}:{PORT}/{DATABASE}
```

字符集：`utf8mb4` / `utf8mb4_unicode_ci`。详见 `WmsSQL/表结构说明.md`。

### 4.2 表清单（13）

`users`、`product`、`warehouse`、`location`、`inventory`、`inventory_transaction`、`inbound_order`、`inbound_order_line`、`outbound_order`、`outbound_order_line`、`stocktake_order`、`stocktake_order_line`、`order_no_seq`

### 4.3 关系概览

```
users ── 创建/审核/操作 ── 各类单据、库存流水

warehouse ── location ── inventory（仓库+库位+编码）
     │
product ──┬── inventory / 流水 / 各单据明细

inbound_order ── line     （入库加库存）
outbound_order ── line    （出库减库存）
stocktake_order ── line   （盘点调库存）

inventory_transaction ← ref_type + ref_id
```

### 4.4 对 AI 的约束

- 禁止绕过 `inventory_service` 改 `inventory.quantity`。
- 单据过账与流水必须在同一事务内完成。
- 密码存哈希；敏感配置只放 `.env`。

## 5. 前后端契约

### 5.1 API 约定

| 项 | 约定 |
|----|------|
| 前缀 | `/api/v1` |
| 格式 | JSON |
| 资源路径 | 复数、kebab-case、无尾斜杠 |
| 分页 | `page`、`per_page`；结果在 `data`：`{ items, page, per_page, total }` |

### 5.2 统一响应体

**成功：**

```json
{
  "success": true,
  "code": "OK",
  "message": "操作成功",
  "data": {}
}
```

**失败（库存不足）：**

```json
{
  "success": false,
  "code": "INSUFFICIENT_STOCK",
  "message": "可用库存不足",
  "data": null
}
```

### 5.3 业务码

| code | HTTP | 说明 |
|------|------|------|
| `OK` | 200/201 | 成功 |
| `VALIDATION_ERROR` | 400/422 | 参数校验失败 |
| `UNAUTHORIZED` | 401 | 未登录 |
| `FORBIDDEN` | 403 | 无权限 |
| `NOT_FOUND` | 404 | 资源不存在 |
| `CONFLICT` | 409 | 单据状态不允许 |
| `INSUFFICIENT_STOCK` | 409 | 出库库存不足 |
| `INTERNAL_ERROR` | 500 | 服务器错误 |

### 5.4 路径维护对照

| 前端 | 后端 |
|------|------|
| `src/api/paths.ts` | `app/routes/*.py` |
| `src/api/*Orders.ts` | `app/services/*_service.py` |
| `src/api/types.ts` | models + 路由序列化字段 |
